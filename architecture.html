1. Overview & Goals

TabFlow is a cross-platform browser tab manager that organizes tabs into workspaces, saves sessions, and syncs across all devices. Think of it as "Workona meets cross-platform sync."

Functional Requirements

P0 Must-Have for MVP

Create, rename, and switch between named workspaces. Each workspace holds a set of tabs. Suspend/restore workspaces to free memory. Save sessions that persist across browser restarts. Cloud sync of workspaces across Chrome, Firefox, Edge, and Safari.

P1 Post-MVP

Tab search across all workspaces. Workspace templates (e.g., "Research" with predefined tabs). Usage analytics (most visited, time tracking). Shared workspaces for teams. iOS/iPadOS companion app.

Non-Functional Requirements

Performance: Workspace switch < 200ms locally, sync latency < 2 seconds between devices.
Storage: Support 50+ workspaces with 100+ tabs each without degradation.
Availability: Offline-first — full functionality without internet, sync when reconnected.
Privacy: URLs/titles encrypted at rest in the cloud. User owns their data.

Key Insight: Shared Core
The Workspace Engine, Sync Client, Storage Adapter, and Auth Manager are written in TypeScript as a shared npm package. Each platform (Chrome extension, Firefox extension, React Native app) imports this core and plugs in platform-specific adapters. This means ~70% of your logic is written once.

Why React Native for mobile instead of a Safari extension?
iOS Safari extensions cannot manage windows, have very limited tab control, and must be packaged inside a native app anyway. A React Native app gives full control over the UX, native push notifications for workspace changes, deep linking to open tabs in Safari, and a proper App Store presence. This mirrors Workona's approach.

7. Sync & Conflict Resolution

Offline-First Architecture

All operations write to local IndexedDB first, then sync to Supabase when online. This ensures the extension works perfectly without internet and feels instant.

Sync flow: User action → write to IndexedDB → update UI immediately → push to Supabase in background → Supabase broadcasts to other devices via Realtime → other devices update IndexedDB + UI.

Conflict Resolution Strategy

Last-Write-Wins (LWW) with field-level merging. Each record has an updated_at timestamp and a version counter. When two devices edit the same workspace:

1. If they edited different fields (e.g., one renamed it, another changed its color), merge both changes.

2. If they edited the same field, the later updated_at wins.

3. Tab ordering conflicts use the version counter — higher version wins.

This is simple, predictable, and good enough for a personal tab manager. Team/shared workspaces (P1) would need operational transforms or CRDTs.

8. Key Architecture Decisions

ADR-1 Why Supabase over Firebase?
Decision: Supabase (PostgreSQL)
PostgreSQL gives relational modeling (workspaces have tabs), flexible querying, and schema evolution via migrations. Supabase's Realtime is built on PostgreSQL LISTEN/NOTIFY with lower latency than Firestore. Open-source with predictable pricing and self-hosting option for the future. Firebase's Firestore document model would require denormalization and makes querying across workspaces harder.
ADR-2 Why a shared TypeScript core?
Decision: Monorepo with shared @tabflow/core package
Writing workspace logic, sync, and storage once and importing it across Chrome, Firefox, Safari, and React Native reduces bugs and development time by ~60%. Platform differences are handled by adapter interfaces (e.g., StorageAdapter wraps IndexedDB on extensions and AsyncStorage on React Native).
ADR-3 Why React Native for mobile instead of PWA?
Decision: React Native
PWAs on iOS lack background sync, push notifications, and App Store distribution. Safari extension on iOS has no window management. React Native provides native UX, push notifications for workspace changes, App Store presence, and shares React knowledge with the extension UI. Flutter was considered but React alignment gives better code sharing with the extension popups.
ADR-4 Why IndexedDB for local storage?
Decision: IndexedDB (via Dexie.js wrapper)
chrome.storage.local has a 10MB limit and is synchronous in MV3 service workers. IndexedDB supports unlimited storage, async access, and indexed queries. Dexie.js provides a clean Promise-based API. On React Native, AsyncStorage serves the same role through the StorageAdapter interface.

9. Development Roadmap

Phase 1 — Weeks 1–4
Chrome Extension + Local Storage
Set up monorepo. Build the shared core (workspace engine, storage adapter). Create Chrome MV3 extension with sidebar UI. Local-only — no cloud sync yet. Workspaces, tabs, save/restore sessions.
Phase 2 — Weeks 5–8
Cloud Sync + Firefox
Set up Supabase project (auth, database, realtime). Build sync client in the shared core. Add user auth (Google OAuth via Supabase). Port to Firefox using WebExtensions compatibility. Offline-first with background sync.
Phase 3 — Weeks 9–12
Safari + Polish
Build Safari Web Extension using Xcode converter tool. Handle Safari API differences via adapter. Add tab search, workspace templates, keyboard shortcuts. Performance optimization and testing.
Phase 4 — Weeks 13–16
React Native Mobile App
Build iOS/iPadOS companion app. Workspace browsing, session management, deep links to open tabs in Safari. Push notifications for workspace changes. App Store submission.

10. Risks & Mitigations

High Manifest V3 Service Worker Limits

MV3 service workers terminate after 30 seconds of inactivity. Tab monitoring must use event-driven patterns (chrome.tabs.onUpdated, chrome.tabs.onRemoved) rather than polling. Mitigation: design all background logic as event handlers, use alarms API for periodic sync.

Medium Safari API Gaps

Safari's WebExtensions implementation lags Chrome. Some APIs (like tabGroups) may behave differently or not exist. Mitigation: build a browser capability detector in the shared core that gracefully degrades features per platform.

Medium Sync Conflicts at Scale

Last-write-wins is simple but could lose data if a user rapidly switches between devices. Mitigation: implement a sync queue with version checking. For MVP this is acceptable; revisit with CRDTs if team workspaces ship.

Low Store Review for Safari/iOS

Apple's review process can reject extensions. Mitigation: follow Apple's Human Interface Guidelines from the start, include proper privacy labels, and test on actual devices before submission.
